# Требования к учебному проекту «Кино вместе»

## 1. Описание системы

Система «Кино вместе» предназначена для совместного просмотра фильмов пользователями через веб-интерфейс. Пользователь может хранить список фильмов «Хочу посмотреть», находить совпадения с друзьями и запускать сеанс совместного просмотра с синхронизированным управлением воспроизведением и встроенным чатом.

Система реализуется как веб-приложение со следующим стеком технологий:

* Backend: FastAPI
* Аутентификация: JWT
* База данных: PostgreSQL
* Хранение видео: MinIO (S3-совместимое хранилище)
* Контейнеризация: Docker / Docker Compose

Система должна запускаться локально и иметь архитектуру, допускающую дальнейшее масштабирование.

---

# 2. Функциональные требования

## 2.1 Управление пользователями

### FR-1 Регистрация пользователя

Система должна предоставлять возможность регистрации пользователя по email и паролю.

Пароль должен храниться в базе данных в хешированном виде.

### FR-2 Авторизация пользователя

Пользователь должен иметь возможность войти в систему по email и паролю.

При успешной авторизации система должна выдавать JWT-токен.

### FR-3 Аутентификация запросов

Все защищённые API должны требовать наличие валидного JWT-токена.

---

# 2.2 Управление друзьями

### FR-4 Добавление друзей

Пользователь должен иметь возможность отправлять запрос на добавление в друзья другому пользователю.

### FR-5 Подтверждение запроса

Пользователь должен иметь возможность принять или отклонить запрос на добавление в друзья.

### FR-6 Просмотр списка друзей

Пользователь должен иметь возможность просматривать список своих друзей.

---

# 2.3 Каталог фильмов

### FR-7 Просмотр каталога

Система должна предоставлять пользователю список доступных фильмов.

### FR-8 Метаданные фильмов

Для каждого фильма должны храниться следующие данные:

* идентификатор
* название
* описание
* длительность
* ссылка на видеофайл в S3-хранилище

---

# 2.4 Список «Хочу посмотреть»

### FR-9 Добавление фильма

Пользователь должен иметь возможность добавить фильм в список «Хочу посмотреть».

### FR-10 Удаление фильма

Пользователь должен иметь возможность удалить фильм из списка.

### FR-11 Просмотр списка

Пользователь должен иметь возможность просматривать свой список «Хочу посмотреть».

---

# 2.5 Поиск совпадений

### FR-12 Проверка совпадений

Система должна предоставлять кнопку «Смотреть вместе».

При нажатии система должна сравнить список «Хочу посмотреть» пользователя со списками его друзей.

### FR-13 Отображение совпадений

Система должна вывести список фильмов, которые присутствуют:

* в списке пользователя
* и в списке хотя бы одного его друга.

### FR-14 Выбор фильма

Пользователь должен иметь возможность выбрать один из совпадающих фильмов для совместного просмотра.

---

# 2.6 Сеанс совместного просмотра

### FR-15 Создание сеанса

Пользователь должен иметь возможность создать сеанс совместного просмотра выбранного фильма.

### FR-16 Приглашение друзей

Пользователь должен иметь возможность пригласить друзей присоединиться к сеансу.

### FR-17 Подключение к сеансу

Приглашённые пользователи должны иметь возможность подключиться к сеансу.

### FR-18 Потоковое воспроизведение

Видео должно загружаться из S3-совместимого хранилища MinIO.

---

# 2.7 Синхронизация воспроизведения

### FR-19 Общие элементы управления

Во время сеанса должны быть доступны следующие действия:

* воспроизведение
* пауза
* перемотка

### FR-20 Синхронизация действий

Действия управления воспроизведением должны синхронно применяться ко всем участникам сеанса.

---

# 2.8 Чат

### FR-21 Чат сеанса

Участники сеанса должны иметь возможность отправлять текстовые сообщения.

### FR-22 Обмен сообщениями в реальном времени

Сообщения должны передаваться между участниками в режиме реального времени.

---

# 3. Нефункциональные требования

## 3.1 Архитектура

### NFR-1 Контейнеризация

Все компоненты системы должны запускаться в Docker-контейнерах.

### NFR-2 Оркестрация

Для локального запуска должен использоваться Docker Compose.

Система должна включать контейнеры:

* backend
* postgres
* minio

---

## 3.2 Производительность

### NFR-3 Время ответа API

Основные API-запросы должны выполняться менее чем за 200 миллисекунд при обычной нагрузке.



---

## 3.3 Безопасность

### NFR-7 Аутентификация

Для аутентификации пользователей должен использоваться JWT.

### NFR-8 Хранение паролей

Пароли должны храниться в базе данных в виде криптографического хеша.
